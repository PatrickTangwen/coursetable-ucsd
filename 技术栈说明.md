# UCSD/SunGrid 与 Upstream CourseTable 技术栈说明

日期：2026-06-27

这份文档记录一次关于 UCSD/SunGrid 当前技术栈、Yale CourseTable upstream 技术栈，以及是否需要迁移到 Ferry/Postgres/Hasura/GraphQL 的讨论。它不是 ADR，也不是迁移计划；用途是帮助后续介绍项目、做技术选型解释、准备面试或设计评审。

## 一句话结论

当前 UCSD/SunGrid 不建议立刻切到 upstream CourseTable 的 Ferry + Postgres catalog + Hasura GraphQL 主路径。

现在更适合的主路径是：

```text
UCSD source pages
  -> tools/catalog-snapshot scripts
  -> api/static/catalogs/public/<TERM>.json
  -> Express/Vite static catalog endpoint
  -> React frontend
```

原因是当前 catalog 数据是公开、只读、快照型数据。核心风险在于能否完整、稳定、可审计地抓取 UCSD 数据，而不是复杂服务端查询。

## 当前 UCSD/SunGrid 技术栈

### Frontend

- React 19
- TypeScript
- Vite
- Zustand store
- React Router
- React Bootstrap / CSS modules
- Catalog 数据通过 REST/static JSON endpoint 读取

当前 catalog 主流程没有依赖 GraphQL。

### Backend

- Node/Bun runtime
- Express API
- TypeScript
- Express session
- Redis session store
- Postgres
- Drizzle ORM
- Docker Compose local backend/auth stack

后端现在主要负责：

- UCSD email auth
- session
- saved worksheets
- saved searches
- serving static catalog JSON

### Data Pipeline

UCSD catalog 数据由 repo 内的 TypeScript scripts 生成，主要在：

```text
tools/catalog-snapshot/
```

它会抓取：

- UCSD Schedule of Classes HTML
- UCSD General Catalog
- UCSD Instructor Grade Archive

并发布：

```text
api/static/catalogs/public/<TERM>.json
api/static/catalogs/import-manifests/<TERM>.json
api/static/metadata.json
```

这些静态 JSON 是当前 UCSD catalog 前端消费的 source of truth。

## 前端有没有用 GraphQL

当前 UCSD/SunGrid 主流程基本没有用 GraphQL。

不用 GraphQL 的路径：

- Catalog list
- Course modal
- Term metadata
- UCSD email login
- Saved worksheets
- Saved searches
- Anonymous worksheet local state

这些分别通过 static JSON、Express REST API、Zustand/local browser state 实现。

repo 中仍有 GraphQL 痕迹：

- `frontend/src/generated/graphql-types.ts`
- `frontend/src/queries/graphql-queries.graphql`
- Apollo / Ferry 相关命名
- `Graphiql` 页面
- backend Hasura/Ferry proxy

这些主要是 upstream Yale CourseTable 遗留，或者用于旧功能/类型兼容。它们不是当前 UCSD catalog 的核心数据路径。

## CourseTable Upstream 那套技术栈如何工作

Upstream Yale CourseTable 更接近：

```text
Yale source data
  -> Ferry crawler / ETL
  -> Postgres course database
  -> Hasura GraphQL
  -> Express static JSON refresh/cache
  -> Frontend
```

它不是单纯每次前端交互都直接打 GraphQL。CourseTable docs 里也说明，Express 会 query GraphQL，再生成 `api/static` 里的 JSON，用来 aggressive cache GQL responses，并支持 client-side course search。

### Ferry

Ferry 是 CourseTable 自己的 Yale course/evaluation crawler。

它负责：

- 从 Yale 原始 source 抓课程和 evaluation 数据
- normalize 成 CourseTable 模型
- 写 reproducible JSON，方便 debug 和重现
- 写入 Postgres course database

可以把 Ferry 理解成 upstream 的长期数据生产系统。

### ETL

ETL = Extract, Transform, Load。

- Extract：从学校课程源抓原始数据
- Transform：把学校原始格式转成 CourseTable 统一模型
- Load：写进数据库或发布产物

Ferry 就承担了 Yale CourseTable 中大部分 catalog/evaluation ETL 责任。

### Postgres

Postgres 是关系型数据库。它适合 CourseTable 这种关系很多的数据：

```text
course
  -> listings
    -> meetings
    -> instructors
  -> evaluations
  -> seasons
```

Postgres 提供：

- 持久化
- 关系查询
- 索引
- schema 演化
- backup / dump
- prod/dev 数据同步

在 upstream CourseTable 中，course database 由 Ferry 写入；user/account database 由 Express API 直接管理。

### Hasura

Hasura 连接 Postgres，并自动暴露 GraphQL API。

CourseTable docs 中的说法是：`graphql-engine` wraps the Postgres database created by Ferry and exposes a GraphQL API。

这让前端或 API 可以写类似：

```graphql
query {
  listings {
    course {
      title
    }
  }
}
```

也可以反向查询：

```graphql
query {
  courses {
    listings {
      crn
    }
  }
}
```

这对关系型课程数据很方便。

### GraphQL

GraphQL 是一种按需查询 API 风格。前端可以声明自己需要哪些字段和关系。

相比 REST:

```text
GET /api/catalog/public/SP26
```

GraphQL 更像：

```graphql
query {
  courses(where: { season_code: { _eq: "202603" } }) {
    title
    listings {
      crn
    }
  }
}
```

它适合：

- 关系型数据查询
- 字段按需返回
- 复杂筛选和关联
- 权限控制
- 对外开放数据查询能力

但它不等于实时数据。底层数据如果是 nightly import，那么 GraphQL 也只是查询“当前数据库状态”，不是 WebReg 级实时源。

## 为什么 CourseTable 要用这套

Yale CourseTable 是一个长期运行的课程平台，不只是一个单 term catalog snapshot。

它要支持：

- 多年、多 term 数据
- course / listing / section / meeting / instructor / evaluation 的复杂关系
- 用户登录
- friends
- evaluation access
- saved user data
- 对外 GraphQL playground / data access
- prod/dev database sync
- 定期 course/evaluation recrawl

这些需求让 Ferry + Postgres + Hasura GraphQL 变得合理。

换句话说，upstream 是一个长期运营的数据平台；不是为了 MVP 简单展示一个 term 的课表。

## 为什么 UCSD/SunGrid 当前没有采用那套

当前 UCSD/SunGrid 的 catalog 主路径是公开、只读、快照型。

现在更重要的是：

- 完整抓取 UCSD Schedule of Classes
- 正确处理分页
- 明确每个 term / subject / source 的成功失败状态
- 保留 raw artifacts 和 import manifest 方便审计
- 让前端稳定消费 snapshot
- 支持 anonymous worksheet 和基础登录后的 saved worksheet

这些需求更适合 file-first snapshot。

当前路径：

```text
UCSD source pages
  -> snapshot scripts
  -> static JSON
  -> frontend local search/filter/sort
```

而不是：

```text
UCSD source pages
  -> DB
  -> GraphQL
  -> per-interaction server query
```

## Static JSON 与 GraphQL 的核心 tradeoff

| 维度          | Static Catalog JSON          | GraphQL / DB-backed    |
| ------------- | ---------------------------- | ---------------------- |
| 数据获取方式  | 预生成快照                   | 按请求查询当前 DB      |
| 前端交互      | 本地 filter/sort/search      | 可能频繁请求后端       |
| 网络依赖      | 首次加载强依赖，之后弱依赖   | 每次查询依赖后端       |
| 更新延迟      | 取决于 snapshot 生成频率     | 取决于 DB 更新频率     |
| 查询灵活性    | 较低，字段预先设计           | 高，按需字段/关系      |
| 实现复杂度    | pipeline 要可靠，服务层简单  | schema/API/DB 复杂     |
| 权限控制      | 不适合敏感细粒度数据         | 更适合权限和角色       |
| Debuggability | raw HTML + manifest 容易审计 | 需要看 DB/schema/query |

对当前 UCSD catalog，Static JSON 的优势更直接。

## 什么时候才值得迁移到 CourseTable 那套

可以等真的需要这些能力时再考虑：

- catalog JSON 体积太大，前端整包加载不可接受
- 需要服务端分页、复杂筛选、复杂排序
- 需要跨 term/course/instructor 的实时组合查询
- 需要后台管理、数据修正、审核工作流
- 需要把 catalog course data 和用户 worksheet/search 做强关联查询
- 需要多学校、多数据源长期运营
- 需要对外提供稳定查询 API
- REST endpoint 已经复杂到难维护

推荐的渐进路线：

1. 保留 snapshot pipeline 作为 source of truth。
2. 前端加 virtualization / lazy loading / index 文件。
3. 如果 JSON 太大，再加轻量 REST search endpoint。
4. 如果 REST search 也复杂，再考虑 DB-backed catalog。
5. GraphQL/Hasura 放最后，不提前引入。

## 本地开发端口约定

本地启动时要同时保证前端和 auth API 都可用。

推荐形态：

```text
API/auth Docker stack: https://localhost:3000
Vite frontend:        https://localhost:3001
Frontend env:         VITE_API_ENDPOINT=https://localhost:3000
```

不要把 Vite 开到 `3000`。`3000` 是 auth API 端口，Vite 抢占或覆盖后会导致 UCSD email login 失败。

如果登录报错，例如：

```text
Failed while requesting ucsd email verification: Error
```

优先检查：

- Docker API/auth stack 是否还在
- 前端是否跑在 `https://localhost:3001`
- API 是否跑在 `https://localhost:3000`
- 前端是否设置了 `VITE_API_ENDPOINT=https://localhost:3000`
- 当前页面是否打开的是 `localhost` 而不是 `127.0.0.1`

## 近期发现：UCSD Schedule of Classes 分页

之前 Spring 2026 CSE 只显示 8 门，是因为 UCSD Schedule of Classes 结果有分页，而 fetcher 只解析第一页。

修复方向：

- `tools/catalog-snapshot/scheduleOfClasses.ts` 读取分页数
- 带 UCSD 返回的 cookie 继续抓 `?page=2/3/...`
- 将所有分页 HTML 合并后交给 parser
- 加 regression test 覆盖多页 Schedule of Classes

修复后本地验证：

```text
SP26 / CSE:
  courses: 88
  sections: 377
```

SP26 全 term schedule source 验证：

```text
schedule_of_classes:
  187 subjects
  187 ok
  0 failed
```

注意：General Catalog enrichment 仍可能有部分 subject 失败，例如 UCSD catalog 页面 404 或跨 subject 页面命名问题。这不等于 Schedule of Classes 课程缺失，只影响 description/prereq 等 enrich metadata。

## 面试/介绍时可以怎么说

可以这样解释：

> Upstream CourseTable 使用 Ferry + Postgres + Hasura GraphQL，是因为它是长期运行的 Yale course data platform，需要多 term、多关系、evaluation、用户功能和对外查询能力。
>
> 但我的 UCSD/SunGrid 当前 catalog 主路径是公开只读的 snapshot 数据。用户会高频 filter/sort/search，如果每次交互都打 GraphQL，会增加请求量、前后端耦合和维护成本。
>
> 所以我先把数据层设计成 Catalog Snapshot：crawler/import/normalization 负责生成按 term 的 JSON 和 manifest；frontend 负责消费 snapshot 做搜索、筛选和 worksheet。
>
> GraphQL/DB-backed catalog 不是不用，而是等到数据规模、查询复杂度或用户态关联需求真的需要时再引入。
